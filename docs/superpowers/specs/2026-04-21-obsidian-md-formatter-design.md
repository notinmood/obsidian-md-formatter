# Obsidian Markdown 格式化插件设计文档

## 概述

一个用于格式化Obsidian中Markdown文件的插件，支持细粒度配置，采用插件式规则架构便于扩展。

## 需求总结

| 维度 | 选择 |
|------|------|
| 格式化范围 | 基础Markdown语法规范化 |
| 运行形式 | Obsidian插件 |
| 触发方式 | 手动触发（命令面板/快捷键） |
| 技术栈 | TypeScript |
| 配置能力 | 详细配置（细粒度控制每条规则） |
| 格式化规则 | 标题、列表、代码块、段落、链接/图片、表格 |
| 错误处理 | 自动修复 |
| 文件编码 | 自动检测常见编码 |
| 大文件处理 | 分块处理避免卡顿 |

## 核心架构

```
┌─────────────────────────────────────────────────────────┐
│                    Obsidian Plugin                       │
├─────────────────────────────────────────────────────────┤
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐     │
│  │   UI Layer   │  │ Config Mgr  │  │  Settings   │     │
│  │  (Commands)  │  │             │  │    Tab      │     │
│  └──────┬──────┘  └──────┬──────┘  └─────────────┘     │
│         │                │                              │
│         ▼                ▼                              │
│  ┌─────────────────────────────────────────────────┐   │
│  │              Formatter Core                      │   │
│  │  ┌───────────┐  ┌───────────┐  ┌───────────┐   │   │
│  │  │  Parser   │→│  Rules    │→│ Renderer  │   │   │
│  │  │ (remark)  │  │ Pipeline  │  │(stringify)│   │   │
│  │  └───────────┘  └───────────┘  └───────────┘   │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 核心组件

1. **UI Layer** - 注册Obsidian命令、快捷键绑定
2. **Config Manager** - 管理用户配置，持久化到Obsidian的data.json
3. **Formatter Core** - 核心格式化引擎
   - **Parser**: 使用remark解析Markdown为AST
   - **Rules Pipeline**: 按配置顺序应用格式化规则
   - **Renderer**: 使用remark-stringify将AST还原为文本

### 数据流

```
原文件内容 → remark解析 → AST → 规则转换 → 新AST → remark-stringify → 格式化后内容
```

## 格式化规则

### 规则配置项

| 规则 | 配置项 | 默认值 |
|------|--------|--------|
| **标题规范化** | 标题前后空行数、是否强制ATX风格 | 前后各1行（一级标题前0行）、true |
| **列表格式化** | 缩进空格数、列表项间距 | 2空格、无额外空行 |
| **代码块处理** | 前后空行数、是否自动补全语言标识 | 前后各1行、true |
| **段落处理** | 段落间空行数、是否清理行尾空白 | 1行、true |
| **链接/图片** | 链接语法风格、引用式链接排序 | inline、false |
| **表格格式化** | 列对齐方式、是否自动调整列宽 | 默认、true |

### 规则执行顺序

```
标题 → 段落 → 列表 → 代码块 → 表格 → 链接/图片
```

顺序可配置，满足依赖关系即可。

## OCP架构设计

采用插件式规则架构，满足开闭原则：

```
┌─────────────────────────────────────────────┐
│             Rule Interface                   │
│  - name: string                              │
│  - priority: number                          │
│  - apply(ast, config): AST                  │
│  - defaultConfig: object                     │
└─────────────────────────────────────────────┘
                    △
                    │ implements
        ┌───────────┼───────────┐
        │           │           │
   ┌────┴────┐ ┌────┴────┐ ┌────┴────┐
   │Heading  │ │ Paragraph│ │  List   │ ...
   │  Rule   │ │   Rule   │ │  Rule   │
   └─────────┘ └──────────┘ └─────────┘
```

### 核心设计

1. **RuleRegistry（规则注册中心）**
   - 管理所有可用规则
   - 支持动态注册/注销规则
   - 按priority排序执行

2. **Rule Interface（规则接口）**

```typescript
interface FormatRule {
  name: string;           // 规则唯一标识
  priority: number;       // 执行优先级
  apply(ast: Node, config: any): Node;  // 转换逻辑
  defaultConfig: object;  // 默认配置
}
```

3. **配置驱动**

```typescript
// 用户配置：启用哪些规则 + 各规则参数
{
  rules: {
    heading: { enabled: true, blankLines: { before: 1, after: 1 } },
    paragraph: { enabled: true, blankLines: 1 },
    // 新增规则只需添加配置项
  }
}
```

### 扩展方式

- 新增规则：实现接口并注册到Registry，无需修改核心代码
- 调整规则：修改配置或替换规则实现
- 禁用规则：配置中设置 `enabled: false`

## 大文件与编码处理

### 大文件分块处理

```
┌─────────────────────────────────────────────────────────┐
│                   File Processor                          │
├─────────────────────────────────────────────────────────┤
│  1. 检测文件大小                                          │
│     - threshold: 500KB（可配置）                          │
│                                                           │
│  2. 小文件 → 正常处理                                      │
│     - 直接解析 → 格式化 → 输出                             │
│                                                           │
│  3. 大文件 → 分块处理                                      │
│     - 边界：标题、代码块、段落等自然分割点                 │
│     - 每块 ≤ chunkSize（默认100KB）                       │
│     - 分块 → 并行处理 → 合并结果                          │
│                                                           │
│  4. 进度提示                                              │
│     - Obsidian Notice显示处理进度                         │
└─────────────────────────────────────────────────────────┘
```

**关键点：**
- 分块边界基于Markdown语义结构，避免破坏语法完整性
- 各块处理完成后按原顺序合并
- 处理过程中显示进度通知

### 编码自动检测

**支持编码：** UTF-8, GBK, GB2312, BIG5

**检测策略：**
1. BOM检测（如有则优先）
2. UTF-8验证（尝试解码，检查是否合法）
3. 常见中文编码探测（基于字符分布统计）

**处理流程：**
```
检测 → 解码为内部UTF-8 → 格式化 → 原编码输出
```

**关键点：**
- 内部统一以UTF-8处理
- 输出时还原为原文件编码
- 检测失败时回退到UTF-8并提示用户

## 配置管理

### 配置存储结构

配置存储在Obsidian标准的 `data.json` 中：

```typescript
interface PluginSettings {
  // 大文件处理
  fileSizeThreshold: number;      // 默认 500KB
  chunkSize: number;              // 默认 100KB

  // 编码检测
  autoDetectEncoding: boolean;    // 默认 true
  fallbackEncoding: string;       // 默认 'utf-8'

  // 规则配置
  rules: {
    [ruleName: string]: {
      enabled: boolean;
      config: object;             // 各规则独立配置
    }
  };
}
```

## 用户交互与错误处理

### 命令注册

| 命令 | 说明 | 快捷键（默认） |
|------|------|---------------|
| `格式化当前文件` | 格式化当前打开的Markdown文件 | `Ctrl/Cmd + Shift + F` |
| `格式化选中内容` | 仅格式化编辑器中选中的文本 | 无 |
| `批量格式化文件夹` | 格式化指定文件夹内所有.md文件 | 无 |

### 处理流程

```
用户触发命令
     │
     ▼
┌─────────────────────────────────────────────┐
│  1. 检查当前文件是否为.md文件               │
│     - 否 → 提示"仅支持Markdown文件"，终止   │
│                                              │
│  2. 检测文件大小                             │
│     - 大文件 → 显示"正在处理大文件..."      │
│                                              │
│  3. 执行格式化                               │
│     - 显示进度通知（大文件时）               │
│     - 分块处理时显示百分比进度               │
│                                              │
│  4. 完成反馈                                 │
│     - 成功 → "格式化完成" + 简要统计         │
│     - 失败 → 显示错误详情                    │
└─────────────────────────────────────────────┘
```

### 错误处理

| 错误场景 | 处理方式 |
|----------|----------|
| 文件编码无法识别 | 提示用户选择编码，或跳过该文件 |
| 解析失败（语法错误） | 尝试定位错误位置，提示用户手动检查 |
| 文件写入失败 | 回滚更改，提示错误原因 |
| 分块处理合并冲突 | 记录冲突位置，提示用户手动处理 |

**关键原则：**
- 不破坏用户原有内容（错误时回滚）
- 提供明确可操作的错误提示
- 大文件处理时保持响应，显示进度
